現在の実装判断では、次を優先する。

## 核となる問い

AIにとって、世界や意味を扱いやすい表現とは何か。

人間の自然言語は、人間の身体・音声・社会生活に適応した表現である。
しかし、AIにとって扱いやすい表現は、それとは違うかもしれない。

この文書は、人間の自然言語を出発点に、言語・記号・意味・媒体・計算機アーキテクチャ・AI表現をつなげて考え、AIにとって自然で実装可能な意味表現の形を探る。

ただし、最終的な基底は人間の自然言語ではなくObservationである。
人間の自然言語は重要な入力源だが、唯一の入力源ではない。

## 言語・記号・トークン

言語は、記号を組み合わせて意味を作る体系である。

記号は何かを表す単位であり、言語は記号を規則に従って組み合わせ、複雑な意味を作る体系である。
すべての言語は記号を使うが、すべての記号が言語であるわけではない。

記号とトークンも区別する必要がある。
記号は意味や機能を担う単位であり、トークンは計算機が処理するために入力を分割した単位である。

```text
記号:
  意味表現の単位に近い

トークン:
  計算機上の入出力単位に近い
```

両者は重なることもあるが、一致するとは限らない。
現在のLLMは、人間の自然言語やコードをトークン列に変換し、その上で統計的・潜在的に意味構造を復元している。

## 媒体と意味構造

言語の本質は音ではない。
音は、人間にとって都合のよい媒体だった。

AIにとって重要なのは、音ではなく、意味・関係・不確実性・状態変化をどう表現し、学習し、更新できるかである。

同じように、表現の表面形式が線形かどうかも本質ではない。
ビット列、音声、文字列、トークン列はいずれも線形に並べられる。
しかし、それらが表す意味構造は線形とは限らない。

人間の自然言語も、表面形式は線形だが、理解される意味は、対象・関係・時間・不確実性・意図・状態変化を含む構造として扱われる。
つまり、自然言語は線形の信号を使って、非線形の意味構造を共有する仕組みと見ることができる。

## 知能一般の自然言語

人間の自然言語は、人間という実装に適応した意味共有形式である。
計算機にとっての自然言語的表現は、計算機という実装に適応した意味共有形式である。

その背後には、知能一般にとっての自然言語という抽象層を考えられる。

それは、特定の媒体やハードウェアではなく、世界状態・信念・目的・行為可能性を共有し更新するための意味表現である。

```text
人間にとっての自然言語:
  人間の身体・音声・文字・社会生活に適応した意味共有形式

知能一般にとっての自然言語:
  世界状態・信念・目的・行為可能性を共有し更新するための意味表現

計算機にとっての自然言語的表現:
  知能一般の意味表現を、計算機上で保持・検索・更新・圧縮・通信しやすくした形式

現代LLMのトークン列:
  人間向け表現やコードを、現在の計算機アーキテクチャで学習しやすくした入出力形式
```

現代の計算機アーキテクチャに由来する制約は重要だが、それは知能一般の自然言語そのものではなく、実装レイヤーの問題として扱うべきである。

## Observation Stream

知能一般や世界モデルを考えるなら、入力を人間の自然言語に限定しない方がよい。
より一般には、入力はObservation Streamである。

```text
Observation Stream
  ├─ human language
  ├─ image / video
  ├─ audio
  ├─ sensor
  ├─ action result
  ├─ tool output
  └─ environment state
```

人間の自然言語は、人間が世界を観測・解釈し、線形テキストに圧縮したものである。
概念、抽象、社会的意味、価値判断、説明、推論パターンを大量に含むため非常に有用である。
しかし、自然言語は世界そのものではない。

```text
世界状態
  ↓ 人間が観測・解釈
自然言語
  ↓ AIが解釈
意味状態
```

一方、画像、センサー、行動ログ、ツール実行結果は、自然言語より低レベルだが、世界により直接近い。

```text
世界状態
  ↓ センサー・観測
非言語データ
  ↓ AIが表現化
意味状態
```

したがって、強いAI設計では、人間の自然言語資産と非言語観測を接続する必要がある。

```text
人間自然言語:
  人間の抽象知識・説明・概念

非言語観測:
  世界との接地・状態変化・行動結果

Semantic State / World State:
  両者を接続する中間表現
```

ClaimやBeliefは自然言語入力と相性がよい。
しかし、ロボットや世界モデルでは、入力は必ずしも主張ではない。

```text
カップが机の端にある
手がカップに接触した
カップが落下した
カップが割れた
```

このような入力では、ClaimよりもEntity、Event、State、StateTransition、Affordance、Risk、CausalHypothesisの方が自然な単位になる。

より一般的なデータモデルは次のようになる。

```text
Observation:
  生の入力。テキスト、画像、音声、センサー、ログなど。

Entity:
  対象。人、物体、概念、場所、システムなど。

Event:
  時間を持つ出来事。

Relation:
  対象同士の関係。

State:
  ある時点の状態。

StateTransition:
  状態の変化。

Hypothesis:
  状態・因果・意味についての仮説。

Belief:
  システムが採用している暫定的理解。

Goal / Preference:
  目的や望ましさ。

Evidence:
  根拠となる観測。

Conflict:
  競合する観測・解釈・仮説。
```

設計の基底には、自然言語ではなく、観測と状態更新を置くべきである。
つまり中心は、`Observation -> StateUpdate -> State` である。

ここでのStateは単一のSemantic Stateではない。
World State、Belief State、Task State、Semantic State、Memory Stateなど、複数種類の状態の束として考える必要がある。
詳細な分類は [notes/state-taxonomy.md](notes/state-taxonomy.md) に分ける。

## 系列と関係

表面形式としての系列と、意味や世界状態の系列は区別する必要がある。

```text
テキスト系列:
  記号やトークンの順序

世界状態系列:
  観測、行為、時間経過による状態の変化
```

人間の自然言語もビット列も、保存や通信の形式としては線形になりうる。
しかし、そこから復元される意味構造や世界状態は、関係、因果、時間変化、不確実性を含む。

したがって、重要なのは系列そのものではなく、系列を通じてどの関係や状態変化を表すかである。

```text
系列:
  関係の一形式

グラフ:
  関係を明示する形式

確率分布:
  不確実性を含む関係の形式

状態更新:
  時間変化を含む関係の形式
```

## 意味表現の最小要件

意味にとって自然な表現は、少なくとも次を扱える必要がある。

```text
対象: 何についての表現か
関係: 対象同士がどう結びつくか
属性: 対象がどんな性質を持つか
時間: いつ成り立つか、どう変化するか
不確実性: どの程度確からしいか
文脈: どの状況・目的・視点での意味か
更新: 新しい観測によってどう変わるか
```

たとえば、「田中が佐藤に本を渡した」は、表面上は文だが、意味としては次のような状態変化を含む。

```text
event: 渡す
actor: 田中
recipient: 佐藤
object: 本
before: 田中が本を持っている
after: 佐藤が本を持っている
confidence: 高い
```

## 表現要素の役割

意味表現は、単一の形式ではなく、複数の表現要素の組み合わせとして考える。

```text
ベクトル:
  類似性、曖昧さ、連想、連続的な意味の近さを扱う。

グラフ:
  対象、関係、参照、構造、因果の向きを扱う。

確率:
  不確実性、複数仮説、信念の強さを扱う。

状態更新:
  時間変化、記憶、観測による意味の変化を扱う。
```

重要なのは、これらが排他的ではないことである。

グラフもベクトル化できるし、確率分布もベクトルとして表せる。
しかし、すべてを同じ密なベクトルに畳み込むと、関係の向き、参照、時間、因果、不確実性が扱いにくくなる。

したがって、表現設計の問題は、どの構造を明示的に扱い、どの構造を潜在表現に任せ、どの操作を安く実行できるようにするかである。

## LLMの位置づけ

現代のLLMは、知能一般にとっての自然言語そのものではない。

それは特に、人間の自然言語と、現代の計算機アーキテクチャの都合に寄った表現システムである。

人間の自然言語に寄っているのは、大量のテキスト、コード、会話、文書、ログが人間向けの記号列として蓄積されてきたからである。
現代の計算機アーキテクチャに寄っているのは、トークン列、ベクトル、テンソル、行列演算が、大規模学習と推論を安く実行しやすいからである。

したがって、LLMは、知能一般の意味表現の最終形というより、人間向けの記号列と潜在的な意味表現を相互変換する強力な近似器として位置づけられる。

得意なのは、曖昧な入力から意味候補を生成することである。
一方で、世界状態を明示的に保持し、厳密に更新し続けることは、LLM単体では扱いにくい。

## Transformerの位置づけ

Transformerは、人間の自然言語専用の構造ではない。
より一般には、トークン化された対象間の関係を、attentionを通じて大規模に計算するアーキテクチャと見られる。

機能としては、トークンを受け取り、トークン間の関係を計算し、関係に基づいて情報を統合し、表現を層ごとに更新する。

attentionは、各層・各ヘッドでトークン間の動的な重み付き関係を作る。

```text
ノード: トークン
エッジ: attention weight
```

ただし、これは明示的な意味グラフではない。
暗黙的・動的・分散的な関係計算であり、永続的な世界状態や信念状態をそのまま保持するものではない。

そのため、Transformerは曖昧な入力から関係候補や意味候補を作るのに強いが、長期的な状態管理、根拠管理、因果更新、不確実性の明示的な更新には、外部の意味状態レイヤーと組み合わせる必要がある。

標準的なself-attentionでは、トークン数を `N` とすると、関係候補数はおおむね `N^2` になる。
これはトークン数そのものが二乗になるという意味ではなく、`N` 個のトークン間で見るべきペア関係候補が二乗で増えるという意味である。

したがって、Transformerの重さは単なる系列長の問題ではなく、関係候補数の問題である。

```text
系列長の問題
  ↓
関係候補数の問題
  ↓
どの関係を計算すべきかの問題
```

理想的には、すべてのペア関係を毎回計算するのではなく、重要な関係だけを選んで計算したい。

## トークン粒度

何をトークンにするかは、単なる前処理ではない。
それは、世界をどの単位に分解し、関係計算の対象にするかという設計問題である。

```text
低レベル:
  文字、バイト、音素、画像パッチ、センサーフレーム

中レベル:
  物体、人物、行為、イベント、場所、関係

高レベル:
  主張、目的、計画、因果関係、信念、状態更新
```

現在のLLMは主にサブワードトークンを使う。
マルチモーダルモデルでは画像パッチ、音声単位、動画の時空間パッチなども使われる。

しかし、知能一般の意味表現を考えるなら、固定粒度だけではなく、低レベル観測から高レベル意味単位までを可変粒度で扱う必要がある。

トークン粒度には、計算量上のトレードオフもある。
細かいトークンに分けると、1トークンは軽くなるが、トークン数が増えて関係候補数が増える。
大きなトークンにまとめると、トークン数は減るが、1トークン内部の情報量と操作コストが増える。

重要なのは、情報を1トークンに詰め込めるかではなく、あとで必要な構造を取り出せるかである。
高粒度トークンは巨大な箱ではなく、圧縮された意味表現、必要な構造、外部メモリへの参照の組み合わせとして考える方がよい。

```text
細かい粒度:
  関係候補が増えやすい

粗い粒度:
  内部表現と展開が重くなりやすい

設計問題:
  どの粒度なら、必要な関係を少ない計算で扱えるか
```

## 実装方向

現実的には、人間の自然言語資産を捨てるのではなく、非言語的な世界モデルや意味状態と接続する方向が強い。

```text
人間の自然言語資産
  ↔
意味状態・世界モデル・行動モデル
  ↔
観測・環境・ツール・データ
```

そのため、いきなり新しい基盤モデルを作るよりも、既存LLMの外側に意味状態レイヤーを置く構成が考えられる。

```text
LLM:
  人間向け表現から意味候補を生成する

Semantic State:
  関係、主張、信念、状態更新を明示的・永続的に保持する

Memory / Retrieval:
  長期知識、根拠、過去の観測を検索する

Update Engine:
  新しい観測によって状態を更新し、矛盾や信念の変化を扱う

Predictive Token Machine / Tool Layer:
  状態変化の予測、行動、外部環境との接続をtyped token stream上で扱う
```

この方向では、局所関係と長距離関係を分けることが重要になる。
近い文脈内の関係はattentionで密に計算し、遠い過去、別文書、別モーダル、抽象概念との関係は、必要なものだけ検索して使う。

```text
その場で計算する関係:
  attention

長く保持すべき関係:
  Semantic State / Graph / Memory
```

外部メモリ付きTransformerやRAGは、文脈制限や長期記憶の弱さを補う方向である。
ただし、多くは文書チャンク、embedding、hidden state、memory tokenを保存するものであり、Entity、Relation、Claim、Belief、Goal、StateUpdateのような高レベル意味状態を明示的に管理するものとは限らない。

Semantic State Memoryは、過去ログそのものではなく、現在の議論や知識がどういう構造になっているかを保存する。

```text
何が重要な概念か
どの主張が有効か
どの主張が修正されたか
何が未解決か
どの関係が残っているか
```

これは単なる要約ではなく、更新可能な意味状態の外部管理である。

## 圧縮・検索・矛盾

Semantic State Memoryで難しいのは、大量の情報をどう圧縮・検索・再利用するか、そして矛盾した入力をどう保持・判定・更新するかである。

基本方針は、すべてを一つの正しい状態に即座に統合しようとしないことである。

```text
生の観測は残す
解釈は別レイヤーにする
信念は確率・根拠・有効範囲つきで持つ
矛盾は削除せず、まず conflict として保持する
必要なときだけ解決する
```

大量情報は少なくとも三層に分ける。

```text
Raw Observation:
  生の入力。元発言、元文書、元ログ。

Semantic Units:
  Entity, Relation, Claim, Event, Question などに分解したもの。

Belief / State:
  現在システムが採用している理解・仮説・状態。
```

重要なのは、Raw Observationを基本的に消さないことである。
解釈は間違う可能性があるが、元の入力が残っていれば後から再解釈できる。

```text
観測:
  immutable

意味状態:
  mutable
```

大量情報を扱うには、分解、重複排除、圧縮、階層化、重要度づけ、検索、状態更新が必要になる。
全情報を毎回Transformerに渡すのではなく、長期保存と作業状態を分ける。

```text
Long-term Store:
  全観測、全claim、全relation、全履歴

Working State:
  今のタスクに必要な少数の意味状態

Context:
  Transformerに渡す選ばれたtoken群
```

矛盾も同じ方針で扱う。
矛盾した入力が来ても古い情報を即座に消さず、Conflictを第一級オブジェクトとして保持する。

```text
Conflict:
  claim_a
  claim_b
  type
  status
  possible_resolutions
```

矛盾には、事実矛盾、時間差による変化、文脈差、視点差、抽象度の違い、解釈ミス、自然言語の曖昧さが含まれる。
そのため、本当に同じ対象について、同じ時間、同じ文脈、同じ抽象度、同じ意味で反対のことを言っているかを確認する必要がある。

Beliefは真偽値だけにしない。

```text
statement
confidence
scope
evidence
counterevidence
status
time
source
```

状態更新にも型が必要である。

```text
add:
  新しいclaimを追加

merge:
  似たclaimを統合

refine:
  既存claimをより正確に修正

contradict:
  既存claimと矛盾するclaimを登録

override:
  新しいclaimが古いclaimを置き換える

retire:
  claimを非アクティブ化する

split:
  一つのclaimを文脈ごとに分ける
```

大量情報と矛盾処理は別問題ではない。
情報が増えるほど、観測源、時点、文脈、抽象度、解釈が増え、矛盾も増える。
したがって、Semantic State Memoryは最初から矛盾を前提に設計する必要がある。

## 人生スケールの意味記憶

知能にとって自然な意味表現を考えるなら、短い文や本1冊だけでなく、人の人生1回分に相当する入力を扱える必要がある。

対象になるのは、長期間にわたる観測、経験、会話、行動、記憶、修正、忘却である。
この規模では、すべてを同じ粒度で保存することはできない。

必要になるのは、単一の現在状態ではなく、多層・多粒度の意味記憶である。

```text
Episodic Memory:
  いつ、どこで、何が起きたか

Semantic Memory:
  世界について一般化された知識

Entity Memory:
  人、場所、物、概念の継続的な同一性

Belief State:
  何をどの程度信じているか

Goal / Preference:
  何を望み、何を避けたいか

Context:
  時期、状況、関係性、視点、社会的文脈

Forgetting / Compression:
  何を保持し、何を要約し、何を捨てるか
```

人生スケールでは、矛盾は例外ではなく通常の入力である。

```text
昔はAを信じていたが、今はBを信じている
同じ人への評価が時期によって変わる
記憶が曖昧になる
後から解釈が変わる
嘘、誤解、比喩、願望、仮定が混ざる
```

したがって、Claimは単に真偽を持つだけでは足りない。

```text
source
time
context
confidence
status
valid_from / valid_until
scope
owner_of_belief
supersedes
conflicts_with
```

特に `owner_of_belief` は重要である。
世界として真なのか、ある人物が信じているのか、過去の自分がそう思っていたのかは区別しなければならない。

人生スケールの意味記憶では、中心問題は「全部覚えること」ではない。

```text
何を現在状態として残すか
何を履歴として残すか
何を抽象化するか
何を忘れるか
何を後から検索できる形にするか
```

## 中心問題

問題は「ベクトルか記号か」ではない。

問題は、

```text
どの構造を明示的に残すか。
どの構造を表現の中に畳み込むか。
どの形式なら、意味を保ちつつAIが安く学習・推論できるか。
```

である。

ここでは、少なくとも次の三つを区別する必要がある。

```text
意味にとって自然な表現
人間の脳にとって安い表現
計算機にとって安い表現
```

この三つは重なるが、一致しない。
人間の自然言語は、人間の身体・音声・社会生活にとって安い表現である。
現代のAIのベクトル表現は、現在の計算機と大規模学習にとって安い表現である。

また、現在のLLMが人間の自然言語を中心に発達しているのは、自然言語がAIにとって本質的に最適な表現だからとは限らない。
大量に存在し、収集しやすく、評価しやすく、人間とのインターフェースとして有用なデータが、主に自然言語として蓄積されてきた影響が大きい。

したがって、AIにとって自然で実装可能な意味表現を考えるには、意味論的忠実性、学習可能性、スケーラビリティ、ハードウェア適合性を同時に見る必要がある。

## 失敗モード

Semantic State Memoryは有望だが、単にEntity、Relation、Claim、Beliefを構造化すれば十分というわけではない。

特に、目的関数、評価、スキーマ進化、忘却、書き込み制御、信頼度の校正、時間スケール、行動による検証、因果、社会的意味、敵対的入力、I/Oコスト、end-to-end学習との関係を扱わないと、ただの構造化メモに留まる可能性がある。

詳細な失敗モードは [notes/failure-modes.md](notes/failure-modes.md) に分ける。

## Bitter Lesson寄りの修正

このプロジェクトは、手設計の知識表現システムに寄りすぎるべきではない。

人間が知能に必要そうな構造を細かく設計し、Entity、Relation、Belief、Goal、Conflict、StateUpdateを固定スキーマとして大量に作ると、最初はきれいに見えても、スケールせず、学習しにくく、硬く、保守しにくいシステムになる危険がある。

したがって、State Memoryは真理のデータベースではなく、観測から作られる派生キャッシュとして扱う。

```text
Raw Observation:
  source of truth

Derived State:
  temporary / revisable cache
```

残すべき最小構造は、細かい意味型ではなく、観測、検索、文脈構成、モデル推論、更新ログである。

```text
Observation Store
  ↓
Retriever / Router
  ↓
Working Context Builder
  ↓
Transformer / LLM
  ↓
Generated Summary / Decision / Next Action
  ↓
Store again
```

この方向では、Claim、Belief、Conflict、Goalなどは、最初から強いDBスキーマとして固定するのではなく、必要なときにモデルが抽出・比較・要約する一時的表現として扱う。
うまく機能した抽象だけをキャッシュし、評価で有用性が確認されたものを残す。

```text
残してよい構造:
  observation
  retrieval
  timestamp
  source
  embedding
  lightweight links
  summaries as cache

慎重にすべき構造:
  fixed ontology
  handcrafted belief system
  manual conflict taxonomy
  hard-coded reasoning rules
  elaborate symbolic state machine
```

つまり、このプロジェクトの核は、細かい意味DBを作ることではない。
LLMが大量の観測履歴から必要な文脈を取り出し、その場で適切な抽象状態を構成できるようにすることである。

## Predictive Token Machineへの位置づけ直し

この節は古い「世界モデル中心」整理を引き継ぐが、現在の上位概念は [Predictive Token Machine](../predictive-token-machine.md) である。
World-modelingはその中核能力の一つとして扱う。

Observation Memoryや外部メモリは主役ではなく、Predictive Token Machineのworld-modeling面を支える補助機構である。

world-modeling面で中心に置くべきなのは、観測と行動から予測に役立つ内部状態を作り、未来や行動結果を予測し、誤差によって状態を更新するループである。

```text
Observation
  ↓
Latent / Predictive State
  ↓
Prediction
  ↓
Action / Query / Tool Use
  ↓
New Observation
  ↓
Prediction Error
  ↓
State Update
```

この観点では、`Semantic State` よりも `Predictive State` が中心に近い。
人間可読なClaimやBeliefは説明・監査・デバッグには便利だが、world-modeling能力を担う表現は潜在状態でよい可能性がある。

外部メモリの役割は、過去観測、キャッシュ、検索基盤として、Predictive Token Machineに渡す情報を選ぶことである。
主題は外部メモリではなく、限られたattention、context、computeの中で、どの情報とどの関係に計算を使うべきかである。

上位概念は [predictive-token-machine.md](../predictive-token-machine.md)、world-modeling面の詳細は [world-model.md](../world-model.md) に分ける。
