# Semantic State Memory Failure Modes

## 目的

Semantic State Memoryを設計する前に、どう壊れるかを明示する。

意味状態を保持し、関係を明示し、矛盾を管理しても、それが何に対して良いのかが定義されていなければ設計は浮く。
この文書は、実装を進める前に見るべき失敗モードを列挙する。

## 目的関数と評価がない

Semantic State Memoryの良さは、目的によって変わる。

```text
回答精度を上げたい
長期議論の一貫性を保ちたい
研究支援をしたい
意思決定を助けたい
ロボット行動を改善したい
```

目的が違えば、必要な状態、更新規則、評価指標も変わる。

失敗モード:

```text
何でも保存するが、何が良くなったのか分からない
構造化は増えるが、タスク性能が上がらない
評価できないため、設計判断が感覚になる
```

## 意味状態の正解が曖昧

自然言語からClaimやRelationを抽出しても、それが本当に正しい分解かは自明ではない。
人間同士でも解釈は割れる。

Semantic Stateは、客観的な意味の写しではなく、ある目的に対する解釈モデルである。

失敗モード:

```text
一つの抽出結果を正解として固定してしまう
別解釈を失う
後から再解釈できない
```

## 明示化しすぎる

Entity、Relation、Claim、Beliefに分けると扱いやすくなる。
しかし現実の意味は、曖昧で、比喩的で、文脈依存である。

明示構造に押し込みすぎると、意味を壊す可能性がある。
暗黙的なベクトル表現が強い場面もある。

失敗モード:

```text
曖昧さを早く潰しすぎる
比喩や含意を落とす
自然言語の柔らかい意味を硬いスキーマに誤変換する
```

## スキーマが進化しない

最初に決めたClaim、Belief、Goal、StateUpdateだけで足りるとは限らない。
議論やタスクが進むと、新しい型が必要になる。

固定スキーマは硬すぎる。
動的スキーマは管理が難しい。

失敗モード:

```text
新しい種類の意味を表せない
無理に既存型へ押し込む
スキーマ変更で過去データが壊れる
```

## 忘却と圧縮がない

人生スケールの入力では、すべてを同じ粒度で保持できない。

```text
何を忘れるか
何を要約するか
何を高粒度トークン化するか
何を元観測として残すか
```

を決める必要がある。

失敗モード:

```text
メモリが増え続ける
検索結果がノイズだらけになる
重要情報が埋もれる
古い情報が不適切に復活する
```

## 書き込みが危険

RAGは主に読むだけだが、Semantic State Memoryは書き込む。
誤った抽出や更新が入ると、以後の推論を汚染する。

必要なもの:

```text
validation
rollback
versioning
provenance
write permissions
```

失敗モード:

```text
誤抽出が長期記憶に入る
誤ったBeliefが後続推論の前提になる
誰が何を書き込んだか追えない
戻せない
```

## 信頼度が校正されない

`confidence: 0.8` が本当に確率的な意味を持つとは限らない。
LLMが出すconfidenceはしばしば当てにならない。

失敗モード:

```text
数値があるため信頼できるように見える
実際には校正されていない
根拠の強さとconfidenceが対応しない
```

## 時間スケールを混同する

複数の更新スケールを分ける必要がある。

```text
短期文脈:
  今の会話

作業状態:
  今のタスク

長期記憶:
  過去の知識・関係

モデル重み:
  学習済みの一般能力

外部世界:
  実際に変化する環境
```

失敗モード:

```text
一時的な文脈を長期記憶に入れる
古い世界状態を現在状態として扱う
モデル重みで覚えるべきことと外部メモリで覚えるべきことを混同する
```

## 行動と検証がない

意味状態を持つだけでは不十分である。
その状態は、検索、実験、ツール実行、環境観測によって検証される必要がある。

失敗モード:

```text
内部的には一貫しているが、外部世界とずれる
検証不能なBeliefが増える
行動結果から更新できない
```

## 因果をRelationと混同する

「AとBが関連する」と「AがBを引き起こす」は違う。
世界モデルを考えるなら、因果、介入、反実仮想を扱う必要がある。

失敗モード:

```text
相関を因果として扱う
介入したときの結果を予測できない
反実仮想を表せない
```

## 社会的・価値的意味がない

自然言語には事実だけでなく、意図、信頼、責任、許可、規範、期待、対人関係が含まれる。

必要になるかもしれない状態:

```text
Commitment
Preference
Norm
Obligation
Trust
Permission
```

失敗モード:

```text
事実Claimだけを扱い、約束や義務を落とす
発話行為を単なる情報として扱う
人間との協働に必要な社会的状態を持てない
```

## 敵対的入力とノイズ

入力は矛盾するだけでなく、意図的に誤らせることもある。

```text
メモリ汚染
曖昧な指示
古い情報の不正な復活
権限のない書き込み
信頼できないソース
```

失敗モード:

```text
攻撃的入力が長期記憶に入る
低信頼ソースが高信頼Beliefを上書きする
権限管理がない
```

## I/Oコストを見落とす

現代計算機のボトルネックはFLOPsだけではない。
外部メモリを使うと、GPU上の高速計算から外に出る。

重要なコスト:

```text
メモリ帯域
キャッシュ
通信
検索レイテンシ
DB更新
分散同期
```

失敗モード:

```text
理論上は良いが遅すぎる
検索やDB更新が支配的になる
分散状態の同期が難しい
```

## End-to-End学習との関係が曖昧

Semantic State Memoryを手設計すると解釈性は上がるが、学習可能性が下がるかもしれない。
全部をニューラルに任せるとスケールしやすいが、制御しにくい。

失敗モード:

```text
手設計が硬すぎて適応できない
学習できる部分が少ない
逆にニューラル化しすぎて制御不能になる
```

## 手設計の知識表現に寄りすぎる

Entity、Relation、Belief、Goal、Conflict、StateUpdateなどを人間が細かく設計しすぎると、The Bitter Lessonが示す失敗に近づく。

失敗モード:

```text
知能に必要そうな構造を大量に手設計する
最初は整って見える
しかしスケールしない
学習しにくい
新しい入力形式に弱い
保守できない
大規模データと計算を使う汎用手法に負ける
```

対策:

```text
Raw Observationを正本にする
Derived Stateはキャッシュとして扱う
固定スキーマは薄くする
抽象化は可能な限りモデルに任せる
評価で有用なキャッシュだけ残す
```

## 意味を状態として固定しすぎる

意味は固定状態というより、文脈、目的、行動、相互作用の中で立ち上がる過程かもしれない。

保存すべきなのは完成した意味状態だけでなく、解釈の生成過程や更新履歴そのものかもしれない。

失敗モード:

```text
意味を静的なDBレコードとして扱いすぎる
解釈過程を失う
目的や行動が変わったときに再解釈できない
```

## まとめ

Semantic State Memoryは有望だが、正解性、スキーマ進化、書き込み制御、忘却、検証、因果、価値、I/Oコストを扱わないと、ただの構造化メモに留まる。

設計原則としては、次を守る。

```text
目的と評価を先に置く
Raw Observationを残す
解釈は複数持てるようにする
Beliefは根拠とスコープつきで更新する
Conflictを第一級オブジェクトにする
書き込みを検証・記録・巻き戻し可能にする
必要な局所状態だけを使う
意味を状態だけでなく過程としても扱う
```
